## Design Notes

### First-Poll Behavior (explicit)

On the very first poll after observer initialization, `window.__phoneCodeLastHTML` is `null`. The early-return condition is:
```js
if (!window.__phoneCodeMutDirty && window.__phoneCodeLastHTML && window.__phoneCodeLastMeta)
```
Because `window.__phoneCodeLastHTML` is `null` (falsy), this condition is **always false on the first poll** â€” the early return is skipped regardless of the dirty flag. The first poll always performs a full DOM clone and capture.

Additionally, the initialization block sets `window.__phoneCodeMutDirty = true` explicitly, providing a second layer of insurance that the first poll is never a cache hit.

### Task 4 â†’ Task 5 Deploy Order Constraint

**Task 5 (client `stats_update` handler) must be deployed before Task 4 (server sends `stats_update`).**

If Task 4 is deployed first, connected clients will receive `{type: "stats_update", stats: {...}}` WebSocket messages. The existing `ws.onmessage` handler has a fallthrough path â€” it will likely log a warning or silently ignore the unknown type (no crash). However, the stats bar will not update and the `âšˇ` indicator will never appear.

**In development** (single server instance, single edit session): apply both Task 4 and Task 5 in the same edit session to avoid any gap.

**In production deployment**: apply `public/js/app.js` (Task 5) first, verify the client handles `stats_update` gracefully, then deploy `server.js` (Task 4).

### Scroll Staleness Trade-off (documented)

`MutationObserver` does not fire on scroll events. If the desktop browser is scrolled while the DOM is idle (no new messages, no mutations), `window.__phoneCodeMutDirty` stays `false` and the phone receives cached HTML with potentially stale `scrollInfo`.

**Why this is acceptable for this tool**: In Claude Code and Antigravity, scroll position changes are almost always coupled with DOM mutations:
- Claude streaming a response: each token fires a `characterData` mutation â†’ dirty â†’ full capture with current scroll
- User navigating conversations: route change remounts the container â†’ dirty
- Static idle state: user not interacting; phone viewport is frozen at last captured position â€” this is the correct behavior (no activity = no update)

The only edge case is a human manually scrolling the desktop during idle without typing anything. For a coding AI tool where the primary use case is reading AI responses (which cause mutations), this is an acceptable limitation. If scroll-only sync becomes a product requirement, add a `scroll` event listener that sets `__phoneCodeMutDirty = true` as a future improvement.

## Verification

1. Start server, let a Claude response complete
2. No new messages â€” DOM settles
3. Next polls: server log shows hash unchanged, no broadcast
4. Server sends `stats_update` with `cached: true`
5. Mobile stats bar shows `âšˇ` (e.g. `"1204 Nodes Â· 38KB Â· âšˇ"`)

### Cache miss â€” streaming

2. Each token fires `characterData` mutation â†’ `__phoneCodeMutDirty = true`
3. Next poll: full clone, `cached: false`
4. Stats bar shows no `âšˇ` during active streaming

### First-poll behavior
1. Server starts cold (no prior snapshot)
2. First poll: `__phoneCodeLastHTML === null` â†’ early return condition fails â†’ full capture
3. `cached: false` in stats
4. No `âšˇ` on first poll ever

2. Observer reconnects to new `cascade` element (isConnected check fires)
3. Next poll performs full capture (no stale HTML)
4. `__phoneCodeObservedEl` updated to new container element

### Cache invalidation
1. Switch target â€” `invalidateSnapshotCache` called
2. Observer disconnected, all 9 globals reset
3. Next poll: full capture (`__phoneCodeMutDirty = true`, `__phoneCodeLastHTML = null`)

### Stats bar sanity
1. `statsText` never shows `NaNKB` or `undefined Nodes` under any path
2. `updateStatsBar` uses `|| 0` guards for all numeric fields
3. `âšˇ` appears only when `stats.cached === true`
